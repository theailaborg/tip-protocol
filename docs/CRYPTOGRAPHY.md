# Cryptography Reference

TIP Protocol mandates post-quantum cryptographic algorithms at the protocol
level. This document specifies the exact algorithms, parameters, and
implementation requirements.

---

## Algorithm Summary

| Function | Algorithm | NIST Standard | Status |
|----------|-----------|--------------|--------|
| Transaction signatures | ML-DSA-65 (Dilithium3) | FIPS 204 | REQUIRED (post-transition) |
| Root/genesis signatures | SLH-DSA-128s (SPHINCS+-SHA2-128s) | FIPS 205 | REQUIRED |
| Key encapsulation | ML-KEM-768 (Kyber-768) | FIPS 203 | REQUIRED for key exchange |
| Hashing | SHAKE-256 (SHA-3 XOF) | FIPS 202 | REQUIRED |
| Legacy (transition only) | Ed25519 | RFC 8032 | PERMITTED (Years 1-3 only) |

---

## Transition Period Policy

During Years 1-3 of the network, **hybrid signatures** are used:

```
signature = concat(Ed25519_sign(msg, ed_key), MLDSA65_sign(msg, pq_key))
```

Both signatures must be present and valid for a transaction to be accepted
during the transition period. After Year 3, Ed25519 is deprecated and only
the ML-DSA-65 signature is required.

Nodes advertise their supported algorithm sets in the gossip handshake,
analogous to TLS cipher suite negotiation.

## Algorithm Details

### ML-DSA-65 (Primary Signatures)

- **NIST FIPS 204**
- Security level: Category 3 (equivalent to AES-192)
- Public key size: 1,952 bytes
- Private key size: 4,032 bytes
- Signature size: 3,309 bytes
- Used for: all identity and content transactions, VP signatures, user signatures

**Production libraries:**
- Node.js: `@noble/post-quantum` (dilithium3)
- Python: `pyoqs` with liboqs, or `pqcrypto`
- Rust: `pqcrypto-dilithium`

**Development stub (current):**
The reference implementation uses Ed25519 as a same-API development stand-in.
This must be replaced before processing real biometric data.
See CHANGELOG.md [BLOCKING-B1].

### SLH-DSA-128s (Root Signatures)

- **NIST FIPS 205**
- Security level: Category 1 (equivalent to AES-128)
- Public key size: 32 bytes
- Private key size: 64 bytes
- Signature size: 7,856 bytes
- Used for: genesis block signing, root of trust anchor

The SLH-DSA root key is the single most security-critical key in the entire
protocol. The genesis block is signed with this key. Key management requirements:

- Stored in a hardware security module (HSM) with FIPS 140-3 Level 3 certification
- Two-of-three custodian policy for key ceremony and use
- Never extractable from the HSM in plaintext
- Air-gapped signing for genesis block only

### ML-KEM-768 (Key Encapsulation)

- **NIST FIPS 203**
- Security level: Category 3 (equivalent to AES-192)
- Public key size: 1,088 bytes
- Ciphertext size: 1,088 bytes
- Used for: encrypted gossip channels (future), session key establishment

### SHAKE-256 (Hashing)

- **NIST FIPS 202**
- Output size: 256 bits (64 hex characters): fixed for deterministic tx IDs
- Used for: tx_id generation, content hashing, TIP-ID generation, dedup hashes

```javascript
// Node.js reference implementation
const { createHash } = require("crypto");
function shake256(data) {
  return createHash("shake256", { outputLength: 32 })
    .update(data)
    .digest("hex");
}
```

## Key Derivation

### TIP-ID Generation

```
tip_id = "tip://id/" + region + "-" + SHAKE-256(ml_dsa_public_key)[:16]
```

The first 16 hex characters of the SHAKE-256 hash of the ML-DSA-65 public key
provide 64 bits of collision resistance: sufficient for a global identity
namespace given the expected network size (< 10 billion identities).

### CTID Generation

```
ctid = "tip://c/" + origin_code + "-" + SHAKE-256(content)[:14] + "-" + tip_id[-4:]
```

The content hash provides content-specific binding. The 4-character TIP-ID
suffix provides author attribution without revealing the full TIP-ID.

### Transaction ID

```
tx_id = SHAKE-256(canonical_json(tx_without_tx_id))
```

Where canonical_json is deterministic JSON serialization (keys sorted
alphabetically, no whitespace).

## Deduplication Hash (v2 Peppered Architecture)

```
pepper       = secure_random(256_bits)  # Generated in device secure enclave
dedup_hash   = SHAKE-256(gov_id || dob || country_code || facial_embedding_hash || pepper)
```

**Pepper properties:**
- 256 bits (32 bytes) of cryptographically random data
- Generated by the device secure enclave at registration time
- Never transmitted to the VP server or the DAG
- Stored only in the device's secure enclave
- If the device is lost, a new TIP-ID can be registered after biometric re-verification

**Why the pepper prevents reidentification:**
Without the pepper, an attacker with access to a government ID database could
attempt to reconstruct the dedup hash for every known person and check against
the DAG. With the pepper, this attack requires also compromising the device's
secure enclave for every individual: making mass reidentification computationally
infeasible.

## ZK Proof of Uniqueness (v2)

The dedup hash is NOT published to the DAG. Instead, a zero-knowledge proof
is published:

```
zk_proof = ZK-SNARK(
  public:   hash_exists_in_registry,
  private:  dedup_hash,
  statement: "I know a value X such that H(X) is in the dedup registry"
)
```

**Production implementation:** snarkjs with Groth16 proving system, or
arkworks with PLONK. The development implementation uses a Pedersen-style
commitment stub (BLOCKING-B2 in CHANGELOG.md).

## Signature Verification

Every DAG transaction must include a signature over the canonical transaction
content. Verification requires:

1. Deserialise the transaction JSON
2. Extract and remove the `signature` field
3. Serialise the remaining fields to canonical JSON (sorted keys, no whitespace)
4. Verify the signature against the signer's known public key

```python
# Python reference
import json
def canonical_json(obj):
    return json.dumps(obj, sort_keys=True, separators=(',', ':'))
```

## ZK Proof System — Identity Deduplication

TIP Protocol uses Groth16 zero-knowledge proofs (snarkjs) with a Poseidon(3) circuit
to enforce one-human-one-identity without storing any biometric or document data on the DAG.

### How it works

```
User's device (VP SDK):
  private inputs:  govId, dob, country   ← never leave the device
        ↓
  Poseidon(govId, dob, country) = dedup_hash
        ↓
  Groth16 proof: "I know inputs that hash to dedup_hash"
        ↓
  { dedup_hash, proof }  ──── sent to node ────►  TIP Node:
                                                    verifies proof with vkey.json
                                                    checks dedup_registry for duplicate
                                                    mints TIP-ID if unique
```

The node never sees `govId`, `dob`, or `country`. The proof is ~200 bytes. Verification takes ~5ms.

### Circuit

**File:** `circuits/dedup.circom`

```
private inputs:  gov_id   (passport/national ID encoded as BN128 field element)
                 dob      (YYYYMMDD integer, e.g. 19900515)
                 country  (ISO-3166-1 alpha-2 encoded as integer)
public output:   dedup_hash  (Poseidon field element, stored in dedup_registry)

constraints: ~264  (Poseidon(3) — ZK-friendly, well within pot18 capacity)
```

### Artifacts

| File | Purpose | Ship to users? |
|---|---|---|
| `circuits/dedup.circom` | Circuit source | In repo |
| `circuits/dedup_js/dedup.wasm` | Compiled circuit — used by SDK to generate proofs | In repo / SDK package |
| `circuits/dedup_final.zkey` | Proving key — used by SDK to generate proofs | In repo / SDK package |
| `circuits/vkey.json` | Verification key — loaded by node to verify proofs | In repo |
| `circuits/*.ptau` | Powers of Tau ceremony file | **Not in repo** (too large — see below) |

### One-Time Setup (run before first launch or after circuit changes)

#### Prerequisites

```bash
# 1. Install circom (Rust binary — required to compile the circuit)
cargo install circom
# or if cargo is not installed: https://rustup.rs

# 2. Install npm dependencies (from the node/ directory)
cd node && npm install
```

#### Step 1 — Download the Powers of Tau file

The ptau file is Phase 1 of the trusted setup. We use the publicly-audited Hermez
ceremony file with 54 contributors. It is not included in the repo (288MB).

Download from the snarkjs GitHub repository:
**https://github.com/iden3/snarkjs** → README → "Prepared (phase2) Ptau files for bn128"

We use: `powersOfTau28_hez_final_18.ptau`

Place it in `circuits/`:

```bash
mv powersOfTau28_hez_final_18.ptau circuits/
```

#### Step 2 — Run the setup script

```bash
node scripts/zk-setup.js --ptau circuits/powersOfTau28_hez_final_18.ptau
```

This performs 4 steps automatically:

**Step 1/4 — Compile circuit**
Runs `circom dedup.circom` to produce:
- `circuits/dedup.r1cs` — constraint system (intermediate, not shipped)
- `circuits/dedup.sym` — symbol file (debug only, not shipped)
- `circuits/dedup_js/dedup.wasm` — WebAssembly circuit (shipped in SDK)

**Step 2/4 — Powers of Tau**
Uses the provided Hermez ptau file directly — skips local generation.
The Hermez file was contributed to by 54 independent participants.
Security property: proof system is secure as long as at least one contributor
deleted their randomness honestly.

**Step 3/4 — Groth16 trusted setup (Phase 2)**
Combines `dedup.r1cs` + ptau → generates the circuit-specific proving key:
- `circuits/dedup_0000.zkey` → contribute randomness → `circuits/dedup_final.zkey`
- Intermediate `dedup_0000.zkey` is deleted automatically

**Step 4/4 — Export verification key**
Extracts `circuits/vkey.json` from `dedup_final.zkey`.
The node loads this at startup to verify every incoming proof.

#### Step 3 — Verify outputs

After setup, you should have:

```
circuits/
  dedup_js/dedup.wasm     ✓  SDK — generates proofs
  dedup_final.zkey        ✓  SDK — generates proofs  (~278KB)
  vkey.json               ✓  Node — verifies proofs  (~3KB)
```

#### Without --ptau (dev/local only)

If no `--ptau` is provided, the script generates a local single-contributor pot12 file.
**Do not use for production** — single contributor means single point of trust.

```bash
node scripts/zk-setup.js   # dev only — single contributor
```

### Runtime

**Client (VP SDK)** — `shared/zk.js → generateDedupProof(govId, dob, country)`
- Loads `dedup.wasm` and `dedup_final.zkey`
- Runs Groth16 proof generation in ~1–3 seconds
- Returns `{ dedup_hash, proof }` — only these are sent to the node

**Server (TIP Node)** — `shared/zk.js → verifyDedupProof(dedupHash, proof)`
- Loads `vkey.json` once at startup
- Verifies Groth16 proof in ~5ms
- Returns `true` / `false`

### Security properties

| Threat | Protection |
|---|---|
| Node learns govId/dob/country | ZK proof — private inputs never leave device |
| Duplicate registration (same person) | `dedup_hash` stored in `dedup_registry` — 409 on duplicate |
| Fake proof (forged) | Groth16 verification against `vkey.json` — computationally infeasible to forge |
| Fake dedup_hash (proof-swap) | Proof is cryptographically bound to `dedup_hash` — cannot reuse proof with different hash |
| Government enumeration | Accepted limitation — issuing gov can compute hashes from own database. See `todo.md` → "Dedup Hash — Pepper" for future mitigation |
